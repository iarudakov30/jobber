# gRPC in Jobber

## Overview

gRPC is used for internal service-to-service communication. Currently `jobs` calls `auth` to validate a JWT token before serving GraphQL requests.

---

## The Contract (`proto/auth.proto`)

```proto
syntax = "proto3";

package auth;

service AuthService {
  rpc Authenticate(AuthenticateRequest) returns (User);
}

message AuthenticateRequest {
  string token = 1;
}

message User {
  int32 id = 1;
  string email = 2;
}
```

The `.proto` file is the single source of truth — it defines the service, methods, and message shapes. It is language-agnostic, so any service in any language (Go, Python, Java) can generate its own client from it.

TypeScript types and decorators (`types/proto/auth.ts`) are generated from this file using `protoc-gen-ts_proto`. Both are needed:

- `auth.proto` — drives binary serialization (Protocol Buffers) and is loaded by NestJS at runtime
- `auth.ts` — provides compile-time safety in TypeScript

---

## Request Flow

```
Client → GraphQL (jobs)
  → GqlAuthGuard extracts Authentication cookie
    → gRPC call to auth: AuthService.Authenticate({ token })
      → JwtAuthGuard validates JWT
      → UsersService.getUser() fetches full user
    ← User returned over gRPC
  → req.user set, guard passes
→ JobsResolver.getJobs() executes
```

## auth — gRPC Server

**`main.ts`** boots both an HTTP server and a gRPC microservice in the same process:

```ts
app.connectMicroservice<GrpcOptions>({
  transport: Transport.GRPC,
  options: {
    package: AUTH_PACKAGE_NAME,
    protoPath: join(__dirname, 'proto/auth.proto'),
  },
});
await app.startAllMicroservices();
```

**`AuthController`** implements the gRPC service. It is registered in `AuthModule.controllers` like any NestJS controller. The gRPC transport discovers it via the `@GrpcMethod` decorator.

```ts
@Controller()
@AuthServiceControllerMethods()
export class AuthController implements AuthServiceController {
  @UseGuards(JwtAuthGuard)
  authenticate(request: AuthenticateRequest & { user: TokenPayload }): Promise<User> {
    return this.usersService.getUser({ id: request.user.userId });
  }
}
```

### `@AuthServiceControllerMethods()` vs `@GrpcMethod()`

`@AuthServiceControllerMethods()` is generated by `protoc-gen-ts_proto`. It applies `@GrpcMethod('AuthService', 'authenticate')` to each method automatically using reflection:

```ts
// generated — do not edit
export function AuthServiceControllerMethods() {
  return function (constructor: Function) {
    const grpcMethods = ['authenticate'];
    for (const method of grpcMethods) {
      const descriptor = Reflect.getOwnPropertyDescriptor(constructor.prototype, method);
      GrpcMethod('AuthService', method)(constructor.prototype[method], method, descriptor);
    }
  };
}
```

The alternative — using `@GrpcMethod()` directly — requires hardcoding magic strings with no compile-time validation:

```ts
// manual — fragile, typos cause silent runtime failures
@GrpcMethod('AuthService', 'authenticate')
authenticate(...) {}
```

Prefer `@AuthServiceControllerMethods()` + `implements AuthServiceController` to keep the proto as the single source of truth.

### One controller per gRPC service

NestJS gRPC maps handlers by `"ServiceName.methodName"`. If two controllers register the same method, the last one wins and silently overwrites the first. This mirrors the gRPC spec — a server has exactly one implementation per service. To split responsibilities, define separate proto services.

### JWT validation over gRPC

`JwtStrategy` accepts the token from either a cookie or `request.token`:

```ts
jwtFromRequest: ExtractJwt.fromExtractors([(request: any) => request.cookies?.Authentication || request.token]);
```

The gRPC transport passes the `token` field from `AuthenticateRequest` directly on the request object, so `request.token` covers the gRPC case without any extra plumbing.

## jobs — gRPC Client

**`JobsModule`** registers the gRPC client pointing at the auth service:

```ts
ClientsModule.register([
  {
    name: AUTH_PACKAGE_NAME,
    transport: Transport.GRPC,
    options: {
      package: AUTH_PACKAGE_NAME,
      protoPath: join(__dirname, 'proto/auth.proto'),
    },
  },
]);
```

**`GqlAuthGuard`** injects the client, gets the typed service proxy on init, and calls it on every protected request:

```ts
onModuleInit() {
  this.authService = this.client.getService<AuthServiceClient>(AUTH_SERVICE_NAME);
}

canActivate(context: ExecutionContext): Observable<boolean> {
  const token = this.getRequest(context).cookies?.Authentication;
  if (!token) return false;

  return this.authService.authenticate({ token }).pipe(
    map((user: User) => {
      this.getRequest(context).user = user;
      return true;
    }),
    catchError(() => of(false))
  );
}
```

The guard returns an `Observable<boolean>` — NestJS resolves it before the resolver runs, making auth fully synchronous from the resolver's perspective.

## Why gRPC and Not Something Else?

| Option                              | Verdict       | Notes                                                            |
| ----------------------------------- | ------------- | ---------------------------------------------------------------- |
| **gRPC**                            | Best fit      | Binary, fast, typed contract, synchronous, NestJS native         |
| **REST/HTTP**                       | Works         | Simpler, but JSON overhead and no shared contract                |
| **GraphQL**                         | Overkill      | Good for client APIs, too heavy for internal calls               |
| **Message queue (RabbitMQ, Kafka)** | Wrong pattern | Async fire-and-forget — auth requires a synchronous response     |
| **NestJS TCP**                      | Acceptable    | Lighter than gRPC, binary, but no proto contract and NestJS-only |
| **Shared Redis session**            | Avoid         | Avoids the network call but creates shared state coupling        |

For synchronous service-to-service calls where both sides are NestJS and speed + type safety matter, gRPC is the right choice.
